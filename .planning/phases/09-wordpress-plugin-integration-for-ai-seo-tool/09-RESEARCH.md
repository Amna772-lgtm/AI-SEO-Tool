## Summary

Phase 9 adds two distinct integration layers. First, a FastAPI API key system (new `api_keys` table, three new auth routes, an extended `get_current_user` dependency that accepts Bearer tokens) so external clients can authenticate without cookie sessions. Second, a WordPress plugin (PHP + React via `@wordpress/scripts`) that lets WordPress site owners connect to their AI SEO Tool account, trigger audits via the existing `/analyze/` endpoint, and view results â€” all from inside the WordPress admin.

The plugin uses a PHP proxy pattern (`register_rest_route` â†’ `wp_remote_post/get`) to call the FastAPI backend server-side, avoiding browser-side CORS entirely. The React layer (built with `@wordpress/scripts`) handles all UI; PHP only handles routing, settings storage (`wp_options`), and proxying. Plan gating is read from `/auth/me` after connection; quota errors (402) surface as upgrade prompts.

The biggest implementation risks are: (1) the CSS scope collision between TailwindCSS v4 and WordPress admin styles â€” use `@wordpress/components` instead to avoid this; (2) the `@wordpress/dependency-extraction-webpack-plugin` that wp-scripts ships requires you to NOT bundle WordPress-provided packages (React, wp-api-fetch, etc.) since WP loads them globally; (3) the `index.asset.php` file generated by the build must be required in PHP before calling `wp_enqueue_script` to get the correct dependency array.

**Primary recommendation:** Build all React UI using `@wordpress/components` (the same library WordPress itself uses in the block editor) â€” this avoids the TailwindCSS/wp-admin CSS conflict entirely and produces a UI that looks native to WordPress admin.

### Pattern 1: wp-scripts Build + PHP Enqueue

**What:** wp-scripts compiles `src/index.js` â†’ `build/index.js` + `build/index.asset.php`. PHP reads the asset file to get correct dependency handles and file hash for cache busting.

**When to use:** Always â€” never hardcode dependency handles or version strings.

**Example:**
```php
// Source: https://developer.wordpress.org/news/2024/03/how-to-use-wordpress-react-components-for-plugin-pages/
function ai_seo_tool_enqueue_admin_scripts( $hook_suffix ) {
    // Only load on our plugin pages
    if ( strpos( $hook_suffix, 'ai-seo-tool' ) === false ) {
        return;
    }
    $asset_file = plugin_dir_path( __FILE__ ) . 'build/index.asset.php';
    if ( ! file_exists( $asset_file ) ) {
        return;
    }
    $asset = include $asset_file;
    wp_enqueue_script(
        'ai-seo-tool-admin',
        plugins_url( 'build/index.js', __FILE__ ),
        $asset['dependencies'],   // auto-detected: react, wp-element, wp-components, wp-api-fetch, etc.
        $asset['version'],        // hash-based; busts cache on rebuild
        array( 'in_footer' => true )
    );
    // Pass PHP â†’ JS: site URL and nonce for WP REST calls
    wp_localize_script( 'ai-seo-tool-admin', 'aiSeoTool', array(
        'siteUrl'  => get_site_url(),
        'apiBase'  => rest_url( 'ai-seo-tool/v1' ),
        'nonce'    => wp_create_nonce( 'wp_rest' ),
        'pluginUrl' => plugins_url( '/', __FILE__ ),
    ) );
}
add_action( 'admin_enqueue_scripts', 'ai_seo_tool_enqueue_admin_scripts' );
```

### Pattern 2: PHP REST Proxy Routes

**What:** PHP registers WP REST routes that forward requests to the FastAPI backend using `wp_remote_*`. The API key is read from `wp_options`, injected into the `Authorization: Bearer` header on every proxied call. Browser never contacts FastAPI directly.

**When to use:** All calls from the WordPress plugin to the AI SEO Tool backend. This is the CORS solution.

**Example:**
```php
// Source: https://yourwpweb.com/2025/09/26/how-to-create-a-proxy-endpoint-for-external-apis-and-avoid-cors-in-wordpress/
add_action( 'rest_api_init', function() {
    // Proxy: POST /analyze/
    register_rest_route( 'ai-seo-tool/v1', '/analyze', array(
        'methods'             => 'POST',
        'callback'            => 'ai_seo_tool_proxy_analyze',
        'permission_callback' => function() { return is_user_logged_in(); },
    ) );
    // Proxy: GET /sites/{id}
    register_rest_route( 'ai-seo-tool/v1', '/sites/(?P<id>[a-zA-Z0-9\-]+)', array(
        'methods'             => 'GET',
        'callback'            => 'ai_seo_tool_proxy_site_status',
        'permission_callback' => function() { return is_user_logged_in(); },
    ) );
    // Auth proxy: GET /auth/me
    register_rest_route( 'ai-seo-tool/v1', '/me', array(
        'methods'             => 'GET',
        'callback'            => 'ai_seo_tool_proxy_me',
        'permission_callback' => function() { return is_user_logged_in(); },
    ) );
} );

function ai_seo_tool_get_backend_url(): string {
    return get_option( 'ai_seo_tool_backend_url', 'https://your-backend.example.com' );
}

function ai_seo_tool_proxy_analyze( WP_REST_Request $request ) {
    $api_key = get_option( 'ai_seo_tool_api_key', '' );
    if ( empty( $api_key ) ) {
        return new WP_REST_Response( array( 'error' => 'Not connected' ), 401 );
    }
    $body    = $request->get_json_params();
    $remote  = wp_remote_post( ai_seo_tool_get_backend_url() . '/analyze/', array(
        'headers'  => array(
            'Content-Type'  => 'application/json',
            'Authorization' => 'Bearer ' . $api_key,
        ),
        'body'    => wp_json_encode( $body ),
        'timeout' => 30,
    ) );
    if ( is_wp_error( $remote ) ) {
        return new WP_REST_Response( array( 'error' => $remote->get_error_message() ), 502 );
    }
    $status = wp_remote_retrieve_response_code( $remote );
    $data   = json_decode( wp_remote_retrieve_body( $remote ), true );
    return new WP_REST_Response( $data, $status );
}
```

**Key implementation note:** The WordPress proxy only needs to proxy these FastAPI endpoints:
- `POST /analyze/` â€” trigger audit
- `GET /sites/{id}` â€” job status (polled every 3 sec)
- `GET /sites/{id}/pages` â€” crawled pages
- `GET /sites/{id}/audit` â€” technical audit results
- `GET /sites/{id}/geo` + sub-routes â€” GEO results
- `GET /auth/me` â€” user/plan info
- `GET /history/` â€” past analyses

### Pattern 3: FastAPI Dual Auth (Cookie + Bearer)

**What:** Extend `get_current_user` in `backend/app/dependencies/auth.py` to accept either the existing HTTP-only cookie JWT OR an `Authorization: Bearer <api-key>` header. Uses `HTTPBearer(auto_error=False)` so missing Bearer does not auto-reject; falls back to cookie check.

**When to use:** All protected FastAPI routes. The existing `Depends(get_current_user)` calls on `analyze.py` and other routes automatically pick up the new behavior with no route changes required.

**Example:**
```python
# Source: FastAPI official security docs + HTTPBearer(auto_error=False) pattern
from fastapi import Cookie, Depends, HTTPException, Request, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from app.store.history_store import get_user_by_id, get_user_by_api_key_hash

_bearer_scheme = HTTPBearer(auto_error=False)

def get_current_user(
    request: Request,
    access_token: str | None = Cookie(default=None),
    bearer: HTTPAuthorizationCredentials | None = Depends(_bearer_scheme),
) -> dict[str, Any]:
    # 1. Try Bearer API key first (used by WordPress plugin)
    if bearer and bearer.scheme.lower() == "bearer":
        raw_key = bearer.credentials
        key_hash = hashlib.sha256(raw_key.encode()).hexdigest()
        user = get_user_by_api_key_hash(key_hash)
        if user:
            # update last_used_at in api_keys table
            update_api_key_last_used(key_hash)
            return user
        raise HTTPException(status_code=401, detail="Invalid API key")
    # 2. Fall back to cookie JWT (existing web browser flow)
    if not access_token:
        raise HTTPException(status_code=401, detail="Not authenticated")
    # ... existing JWT decode logic ...
```

### Pattern 5: Job Status Polling in React (WordPress context)

**What:** Recursive `setTimeout` hook that polls the WP REST proxy every 3 seconds until status is `completed` or `failed`. Uses `@wordpress/api-fetch` (not raw `fetch`) since it auto-applies the WP REST nonce.

**When to use:** All job status polling in the WordPress plugin React code.

**Example:**
```javascript
// Source: https://overreacted.io/making-setinterval-declarative-with-react-hooks/
import apiFetch from '@wordpress/api-fetch';
import { useState, useEffect, useRef } from '@wordpress/element';

function useAnalysisPoller(siteId, intervalMs = 3000) {
    const [status, setStatus] = useState(null);
    const [data, setData] = useState(null);
    const timeoutRef = useRef(null);
    const activeRef = useRef(true);

    useEffect(() => {
        if (!siteId) return;
        activeRef.current = true;

        const poll = async () => {
            try {
                const result = await apiFetch({
                    path: `/ai-seo-tool/v1/sites/${siteId}`,
                });
                if (!activeRef.current) return;
                setStatus(result.status);
                setData(result);
                if (result.status !== 'completed' && result.status !== 'failed') {
                    timeoutRef.current = setTimeout(poll, intervalMs);
                }
            } catch (err) {
                if (activeRef.current) {
                    timeoutRef.current = setTimeout(poll, intervalMs * 2); // back-off on error
                }
            }
        };
        poll();
        return () => {
            activeRef.current = false;
            clearTimeout(timeoutRef.current);
        };
    }, [siteId, intervalMs]);

    return { status, data };
}
```

### Pattern 7: Next.js API Keys Settings Page

**What:** New route `frontend/app/dashboard/settings/api-keys/page.tsx` (or similar path within the existing dashboard layout). Follows the existing `admin` page pattern: `"use client"` component with `useAuth()` guard, calls `apiFetch()` to the three new `/auth/api-key*` endpoints.

**UX:** Generate button â†’ inline display of raw key (one-time) â†’ list of named keys with revoke button. Same TailwindCSS v4 styling as rest of the dashboard.

### Anti-Patterns to Avoid

- **Bundling WP packages into build.js:** `@wordpress/dependency-extraction-webpack-plugin` (included in wp-scripts) externalizes React, ReactDOM, `@wordpress/element`, `@wordpress/components`, etc. They're loaded from WordPress globals. Never import them via npm bundle â€” causes duplicate React instance errors.
- **Calling FastAPI directly from browser JS:** The entire point of the PHP proxy is to avoid this. React code calls `/wp-json/ai-seo-tool/v1/...` only.
- **Activating plugin before setting API key:** Activation hook must NOT make any external calls (WordPress.org rule D-19). Store backend URL and API key in `wp_options`, which the user sets post-activation.
- **Using `setInterval` for polling:** Use recursive `setTimeout` instead (see Pattern 5). `setInterval` stacks calls under slow network.
- **Tailwind v4 in WP admin:** WordPress admin CSS is unlayered and wins the cascade over Tailwind's layer-based rules. Use `@wordpress/components` for plugin UI instead.
- **Storing raw API key in `wp_options`:** The raw key is already a shared secret the user copied from the AI SEO Tool dashboard. Storing it plaintext in `wp_options` is acceptable (same as any API key in wp-config.php). Do NOT re-hash it in WordPress â€” the PHP proxy needs to send the raw key in the Authorization header.
- **Using `manage_options` for non-admin WP users:** The plugin targets site owners/admins. `manage_options` capability is correct; don't lower it to `read`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WP script dependency resolution | Manual dependency array | `build/index.asset.php` from wp-scripts | Auto-detects all `@wordpress/*` imports; fails silently if wrong version loaded |
| Authenticated REST calls from React | Raw `fetch()` with manual nonce | `@wordpress/api-fetch` | Nonce auto-applied; handles nonce refresh; consistent error handling |
| HTTP proxy to external API | PHP cURL directly | `wp_remote_post/get` | WP proxy support, SSL config, error wrapping with `is_wp_error()` |
| API key generation | UUID or md5 | `secrets.token_urlsafe(32)` | Cryptographically secure; URL-safe base64 format |
| API key hashing | bcrypt (too slow for lookup) | `hashlib.sha256` | O(1) lookup by hash; consistent with session token pattern in codebase |
| Bearer token extraction in FastAPI | Manual header parsing | `HTTPBearer(auto_error=False)` | Integrates with FastAPI OpenAPI docs; handles scheme validation |
| WP menu rendering markup | Custom HTML page | `add_menu_page` + `add_submenu_page` | WP admin menu system handles highlight states, capability checks, page routing |

**Key insight:** The WordPress plugin ecosystem has solved the hardest parts â€” script dependency resolution, nonce-based REST auth, proxy HTTP clients, and admin menu management. Building custom solutions for these introduces subtle bugs (wrong React version, failed nonce refresh, SSL failures, broken menu highlight states) that are hard to debug.

## Common Pitfalls

### Pitfall 1: Missing `index.asset.php` File
**What goes wrong:** `wp_enqueue_script` called with hardcoded empty dependency array `[]`. Script loads before React is available; crashes with `React is not defined`.
**Why it happens:** Developer forgets to run `npm run build` before testing, or commits `src/` but not `build/`.
**How to avoid:** In PHP, check `file_exists($asset_file)` before including. Return early if missing. Gate on having run build. Add `build/` directory to git (WordPress.org uses SVN but git repo should track the build output).
**Warning signs:** Browser console shows `React is not defined` or `wp.element is not defined`.

### Pitfall 2: Duplicate React Instance
**What goes wrong:** Plugin bundles React via npm instead of using WordPress's global. Two React instances coexist; hooks break with "Invalid hook call" errors.
**Why it happens:** Developer imports `react` directly in `src/index.js` instead of `@wordpress/element`.
**How to avoid:** Always import from `@wordpress/element`: `import { useState, useEffect } from '@wordpress/element'`. The dependency extraction plugin in wp-scripts handles this automatically â€” it will not bundle `react` into `build/index.js`.
**Warning signs:** "Invalid hook call" or "Hooks can only be called inside a React function component" in WP admin.

### Pitfall 3: WordPress.org Rejection â€” External Call on Activation
**What goes wrong:** Plugin uses `register_activation_hook` to call the FastAPI backend to validate the API key. WordPress.org reviewers reject the plugin.
**Why it happens:** WP.org rule: activation hooks must not contact external servers. Consent requires explicit user action post-activation.
**How to avoid:** `register_activation_hook` only creates/checks database tables or sets default `wp_options`. Connection to the backend is initiated only when the user clicks "Connect" in the plugin settings screen.
**Warning signs:** Plugin review feedback mentioning "external calls during activation."

### Pitfall 4: CSRF Exposure on PHP Proxy
**What goes wrong:** PHP proxy `permission_callback` returns `true` (allows anyone), enabling CSRF attacks where a malicious page makes a cross-origin POST to the proxy, which forwards it to the backend with the stored API key.
**Why it happens:** Developer copies the public-proxy example and doesn't restrict access.
**How to avoid:** Always use `permission_callback` that checks `is_user_logged_in()` AND verify the WP REST nonce. The `@wordpress/api-fetch` library in the React code auto-sends the nonce; PHP REST API validates it before calling the callback.
**Warning signs:** Proxy routes accessible to unauthenticated users in browser dev tools.

### Pitfall 5: Tailwind v4 CSS Cascade Conflict
**What goes wrong:** Plugin uses TailwindCSS v4. WordPress admin CSS is unlayered; Tailwind v4 uses `@layer utilities`. Unlayered CSS has higher specificity than any layer â€” WP admin styles override Tailwind classes. Buttons, inputs, and layout look broken.
**Why it happens:** TailwindCSS v4 introduced CSS cascade layers, which is incompatible with older unlayered CSS like WordPress admin.
**How to avoid:** Use `@wordpress/components` for all plugin UI. They share the same style tokens as WP admin and never conflict.
**Warning signs:** Styled elements in WP admin look unstyled or override-styled with WP defaults.

### Pitfall 6: wp_remote_get Timeout on Large Crawls
**What goes wrong:** Plugin triggers a crawl via the PHP proxy. `wp_remote_post` has a default timeout of 5 seconds. The `/analyze/` endpoint validates robots.txt before returning (can take 2-5s). Request times out; plugin shows "Connection failed."
**Why it happens:** `wp_remote_*` default timeout is 5 seconds.
**How to avoid:** Set `'timeout' => 30` on all `wp_remote_*` calls in the proxy. The `/analyze/` endpoint returns immediately after dispatching the Celery task â€” the long crawl runs in background â€” so 30s is more than enough.
**Warning signs:** Intermittent "Connection failed" on fast backends; 504 responses from the proxy.

### Pitfall 7: API Key Shown Multiple Times
**What goes wrong:** Frontend shows the raw API key every time the user visits the API Keys page (by storing it in the database or returning it from GET).
**Why it happens:** Developer stores the raw key to allow re-display.
**How to avoid:** `POST /auth/api-key` returns the raw key ONCE in the response. The database stores only `key_hash`. `GET /auth/api-keys` returns id, name, created_at, last_used_at â€” never the raw key or hash. Frontend shows a one-time modal on generation; subsequent views show `â€˘â€˘â€˘â€˘â€˘â€˘â€˘â€˘â€˘â€˘â€˘â€˘` placeholder.

## Open Questions

1. **Backend URL configuration in the WordPress plugin**
   - What we know: The plugin proxy needs to know the FastAPI backend URL (e.g., `https://api.yourdomain.com`). This must be user-configurable.
   - What's unclear: Should this be a hard-coded default in the plugin, or a required field in the Settings screen? If hard-coded, it only works for one deployment.
   - Recommendation: Add a "Backend URL" field in the WordPress plugin Settings tab. Default to a placeholder. The user sets it along with the API key. Alternatively, the backend URL is embedded in the API key metadata and returned from `/auth/api-key` â€” then the plugin auto-configures it.

2. **Plan gating endpoint: `/auth/me` vs new `/auth/plan`**
   - What we know: D-15 says detect plan from `/auth/me` or a new `/auth/plan`. The existing `/auth/me` returns `{id, email, name, is_admin}` â€” no `plan` field.
   - What's unclear: Does the planner add `plan` to `/auth/me`, or create a new endpoint?
   - Recommendation: Extend `/auth/me` to include subscription data (plan, audit_count, audit_limit). This is one call instead of two, and existing web clients benefit too. No separate `/auth/plan` endpoint needed.

3. **WordPress.org SVN structure vs Git**
   - What we know: WordPress.org uses SVN for plugin hosting. The `trunk/`, `tags/`, and `assets/` directories are required.
   - What's unclear: The project uses Git. The plan needs a step for SVN submission setup.
   - Recommendation: Keep development in `wordpress-plugin/` in the main Git repo. For WP.org submission, use a GitHub Action or manual SVN push. This is a deployment concern, not a build concern â€” the plan should note it but not block Phase 9 completion.

4. **Polling when WP REST nonce expires**
   - What we know: WP REST nonces expire after 24 hours by default. `@wordpress/api-fetch` auto-refreshes nonces on 403 responses.
   - What's unclear: Whether the auto-refresh behavior works correctly when the React app has been open for many hours (e.g., user left the audit running overnight).
   - Recommendation: The `@wordpress/api-fetch` nonce middleware handles this automatically (HIGH confidence â€” documented in WP developer blog). No custom handling needed.
